# Accessibility

`@ds/core` commits to **WCAG 2.2 AAA where feasible, AA always**. This is not a checklist we
audit at the end — it's a contract every primitive enforces by construction.

The full list of obligations is in `CONTRACT.md §6`. This doc explains how each one is met and how
to verify.

---

## 1 · Color contrast — AAA on body, AA on large text

| Surface pair | Min ratio (WCAG) | Our floor |
|---|---|---|
| Body text on background | 4.5:1 (AA) | **7:1 (AAA)** |
| Large text (≥18px or ≥14px bold) on background | 3:1 (AA) | **4.5:1 (AAA)** |
| Non-text UI (focus ring, icons) | 3:1 (AA) | **3:1 (AA)** |

Every semantic text/background pair in every theme is verified against these thresholds. The
verifier lives in `packages/ds/src/themes/__tests__/contrast.test.ts` and reads the active
theme's resolved tokens, computing oklch → APCA contrast.

### How to verify locally

```bash
# Headless contrast check across all 8 themes
pnpm --filter @ds/core test contrast

# Visual contrast across all themes via axe-core in browser
pnpm dev:examples           # serves examples/ at :4321
# then in the browser console:
# > axe.run(document, { runOnly: ['color-contrast'] }).then(console.log)
```

### Caveat — `glass` theme

The `glass` theme renders text on a translucent surface whose contrast depends on what's behind.
The theme **assumes a sufficiently dark backdrop** and ships a `[data-ds-theme="glass"]
*::backdrop` rule that enforces a minimum darkness via a `mix-blend-mode: multiply` overlay.
Don't apply `glass` on light photographs without testing.

## 2 · Keyboard navigation

Every interactive primitive supports:

| Key | Action |
|---|---|
| `Tab` / `Shift+Tab` | Move focus across tab stops |
| `Enter` / `Space` | Activate primary action |
| `Esc` | Close menu, modal, popover, drawer, sheet |
| `↑` `↓` `←` `→` | Roving focus within lists, menus, tabs, sliders, calendars |
| `Home` / `End` | Jump to first / last item |
| `PageUp` / `PageDown` | Jump by 10 in long lists, calendars |
| `Type-ahead` | First-letter search inside dropdown, select, command palette |

Modals, drawers, sheets and the command palette **trap focus** while open and **restore focus**
to the trigger on close.

### Verify

```bash
# Keyboard navigation lint via @testing-library + user-event
pnpm --filter @ds/core test keyboard

# Manual: unplug your mouse. If you can't reach a feature, it's broken.
```

## 3 · Screen reader semantics

Built on **Radix UI primitives** for behavior, augmented with explicit `aria-*` where Radix
leaves it open. The matrix:

| Primitive | Role / ARIA |
|---|---|
| `Button` | `button`, `aria-busy` when loading, `aria-disabled` when disabled |
| `IconButton` | `button`, **`aria-label` required** at type level |
| `Input` / `Textarea` | label wired via `<label for="…">` or `aria-labelledby`; `aria-invalid` + `aria-describedby` on error |
| `Select` / `Combobox` | `combobox` + `listbox` + `option` per ARIA Authoring Practices |
| `Checkbox` / `Switch` | `checkbox` / `switch`, `aria-checked="true|false|mixed"` |
| `Radio` | `radio` inside `radiogroup` |
| `Modal` / `Drawer` / `Sheet` | `dialog`, `aria-modal="true"`, `aria-labelledby` to title |
| `Dropdown` / `ContextMenu` | `menu` + `menuitem`, `aria-expanded` on trigger |
| `Tabs` | `tablist` / `tab` / `tabpanel`, `aria-controls` + `aria-selected` |
| `Tooltip` | `tooltip`, paired with `aria-describedby` on the trigger |
| `Popover` | `dialog` for interactive, `region` for informational |
| `Toast` | `status` for info/success, `alert` for warning/error |
| `Table` / `DataGrid` | `table`, `row`, `columnheader`, `aria-sort`, `aria-rowcount`, `aria-rowindex` |
| `Pagination` | `navigation` with `aria-label="Pagination"` |
| `Breadcrumb` | `navigation` with `aria-label="Breadcrumb"` + `aria-current="page"` |
| `Badge` | role inferred from semantics — purely decorative or `status` for live updates |
| `Avatar` | `img` with `aria-label`, falls back to text for initials |
| `Loader` | `progressbar` if determinate, `status` + `aria-live="polite"` if indeterminate |
| `Skeleton` | `aria-busy="true"` on the container, hidden from AT once loaded |

### Verify

```bash
# Tested with VoiceOver (macOS), NVDA (Windows), TalkBack (Android)
# We document expected announcement strings per primitive in:
# packages/ds/src/primitives/<name>/<name>.test.tsx → "announces …" cases
```

## 4 · `prefers-reduced-motion`

A single CSS rule in `src/motion/reduced.css` collapses every duration token to `1ms` when the
user has enabled reduce-motion at the OS level:

```css
@media (prefers-reduced-motion: reduce) {
  :root {
    --ds-motion-duration-fast:       1ms;
    --ds-motion-duration-base:       1ms;
    --ds-motion-duration-moderate:   1ms;
    --ds-motion-duration-slow:       1ms;
    --ds-motion-duration-deliberate: 1ms;
  }
  *,
  *::before,
  *::after {
    animation-duration: 1ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 1ms !important;
    scroll-behavior: auto !important;
  }
}
```

Decorative motion (loaders, sparkles) is **removed**, not just shortened. The motion engine in
`src/motion/` checks the `prefers-reduced-motion` media query and conditionally returns no-op
variants for non-essential animations.

### Verify

- macOS: System Settings → Accessibility → Display → Reduce motion → ON
- Windows: Settings → Accessibility → Visual effects → Animation effects → OFF
- Then reload an example app; modal open should be instant, no scale-in.

## 5 · Touch targets ≥ 44×44px

WCAG 2.5.5 floor. All interactive primitives meet `var(--ds-spacing-11)` (44px) on their hit area
even when the visual is smaller (icon-buttons, small toggles).

- A small `IconButton` is rendered with `min-height: var(--ds-spacing-11)` even if the icon is
  20px — the extra padding is invisible but tappable.
- Switch / Checkbox / Radio hit areas extend to the parent label via `<label>` wrap.

### Verify

```bash
# Lighthouse audit
# > Lighthouse → Accessibility → "Touch targets are sized appropriately"
```

## 6 · Focus rings

Every interactive element shows a **visible `:focus-visible` ring** using `--ds-color-focus-ring`,
2-3px thick. Never `outline: none` without an alternative.

```css
.ds-button:focus-visible {
  outline: 2px solid var(--ds-color-focus-ring);
  outline-offset: 2px;
}
```

In `forced-colors` mode (Windows High Contrast), the ring is replaced with `CanvasText` so it
remains visible against the system color override:

```css
@media (forced-colors: active) {
  .ds-button:focus-visible {
    outline: 2px solid CanvasText;
  }
}
```

## 7 · Forced-colors / Windows High Contrast

All primitives ship a `@media (forced-colors: active)` block that:

- Replaces token-based colors with system keywords (`CanvasText`, `Highlight`, `LinkText`, `Mark`,
  `GrayText`, `ButtonText`, `ButtonFace`, `Field`, `HighlightText`).
- Removes decorative gradients / shadows.
- Keeps borders visible on otherwise-flat surfaces by setting `border: 1px solid CanvasText`.

The matrix lives in `packages/ds/src/themes/forced-colors.css`. It applies on top of every theme.

### Verify

- Windows: Settings → Accessibility → Contrast themes → pick any theme → reload example.
- Chrome DevTools: Rendering panel → "Emulate CSS media feature forced-colors" → `active`.

## 8 · Language and reading order

- Components don't break the DOM reading order (no `tabindex > 0`).
- Visually-only re-ordering uses `order` / grid placement; the DOM stays linear.
- Headings are nested correctly per page (H1 → H2 → H3, no skips).
- `lang` attribute is honored — date/time/number primitives format via `Intl` with the active
  language.

## 9 · QA checklist (per release)

Run before tagging any version of `@ds/core`. Stored as
`packages/ds/scripts/a11y-release-check.sh`.

- [ ] `pnpm --filter @ds/core test contrast` — passes across all 8 themes
- [ ] `pnpm --filter @ds/core test keyboard` — all primitives navigable
- [ ] `pnpm --filter @ds/core test a11y` — axe-core zero violations on examples
- [ ] Manual VoiceOver pass on `examples/dashboard.tsx`
- [ ] Manual NVDA pass on `examples/ai-saas-dashboard.tsx`
- [ ] Reduce-motion toggle: no animation visible on modal open / toast slide-in
- [ ] Forced-colors mode: every surface keeps a visible border + readable text
- [ ] Lighthouse accessibility ≥ 95 on each example
- [ ] Touch target audit: zero violations
- [ ] Color-vision-deficiency: deuteranopia / protanopia / tritanopia previews
      (Chrome DevTools → Rendering → "Emulate vision deficiencies") — UI remains
      readable on every theme

A failed check is a blocking release issue, not a warning.

## 11 · When AAA isn't feasible

A handful of designs can't hit AAA on contrast — typically:

- Premium-feel surfaces (e.g. `luxury` theme gold accent on a warm dark background hits ≈4.8:1
  on body text, which is AA but not AAA at 7:1).
- Decorative captions inside images or hero panels.

When this happens:

1. We document the exception in the theme's README (`src/themes/<id>/README.md`).
2. We never drop below AA — that floor is hard.
3. We offer the user a high-contrast variant (e.g. `enterprise`) that always hits AAA.

The user can switch to it via the OS preference or the in-app theme switcher.
