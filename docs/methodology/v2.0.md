# Hearst Yield Vault — Methodology v2.0 (Monte Carlo extension)

**Status**: Active
**Effective from**: 2026-05-22
**Owner**: Product + Risk
**Supersedes for MC mode**: extends v1.0 (rule-based stays the immutable reference)
**ADR**: ADR-006

## Scope

v2.0 **adds a Monte Carlo (MC) simulation mode** to the projection engine. It does
**not** replace v1.0: rule-based projection remains the **default** and the
immutable reference. MC is an *optional* mode the user opts into for a
probabilistic view. Every input source, the APY-range rule, the confidence score,
the forbidden-words list, and the disclaimers from v1.0 carry over unchanged.

This document is itself immutable once published. Any change requires `v2.1`/`v3.0`
and a new ADR.

## Inputs

Identical to v1.0 (BTC price, difficulty, hashprice, energy cost, uptime, stable
APY, T-bills, RWA). MC adds **per-variable volatility + correlation assumptions**,
estimated from the trailing 36-month `MiningMetric` history:

| Variable | Stochastic treatment | Calibration |
|---|---|---|
| BTC price | Geometric Brownian motion (drift μ, vol σ) | σ from 36m daily log-returns |
| Network difficulty | Mean-reverting drift, bounded step | from mempool adjustment cadence |
| Hashprice | **Derived** from (BTC, difficulty) via shared formula — never sampled independently | n/a |
| Energy cost | Fixed contractual + small spot noise | contract band |
| Stable / T-bill APY | Low-vol normal around trailing mean | 36m |

Hashprice is **never** an independent random variable — it is recomputed from the
sampled (BTC, difficulty) each path via `hashprice-formula.ts`, exactly as in live
and backfill. This keeps MC consistent with the rest of the system.

## Determinism (engine purity — non-negotiable #6)

- The PRNG **seed is an explicit engine input**. Same seed + same assumptions ⇒
  byte-identical output. No `Math.random()`, no `Date.now()` inside the engine.
- Default path count: **10,000**. Configurable; the seed and N are recorded in the
  output so any run is reproducible and snapshot-testable.

## Output

```
For each of N paths:
  sample BTC, difficulty (correlated) over the horizon
  derive hashprice → mining yield → blended portfolio yield
  → path_apy
Collect { path_apy } → empirical distribution
```

Reported:

- **Headline stays a range** (non-negotiable #1): `apy_low = p25`, `apy_high = p75`
  of the MC distribution. **No single-point APY is ever published**, MC or not.
- MC **adds** a distribution summary: **p5 / p50 / p95** and the probability of
  finishing below a stated floor (e.g. `P(apy < 8%)`). These are clearly labelled
  as model output, never as a forecast.
- Provenance badge on the MC block = `estimated` (model), with the seed + N + the
  methodology link surfaced.

## APY range vs rule-based

The rule-based v1.0 range (`projected_apy × (1 ± factor)`) and the MC range
(`[p25, p75]`) are shown **side by side** when MC mode is on, so the LP sees both
the deterministic band and the probabilistic spread. They are not blended.

## Confidence score

Unchanged from v1.0. MC additionally degrades confidence when the calibration
window has gaps or when sampled paths drift outside the 36m P5–P95 historical
envelope (out-of-range assumptions weight applies to the *calibration*, not each path).

## Forbidden in any MC output

Everything in v1.0 §"Forbidden in any projection output", plus:

- Presenting a percentile (p50, etc.) **without** the interval and a link to this
  methodology.
- Implying the distribution is a forecast of actual returns.
- Numeric promises, "risk-free", "guarantee", "will deliver", "certain" (#5).
