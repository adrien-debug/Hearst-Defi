## 1. Versioning & Immutability

### 1.1 Version lineage

| Version | Status | Effective | Notes |
|---|---|---|---|
| `v1.0` | **Active / Immutable** | 2026-05-13 | Rule-based projection, single vault |

### 1.2 Semver convention

Versions follow `MAJOR.MINOR` only (no patch):

- **MAJOR bump** (`v2.0 → v3.0`): a formula or model change that makes projections
  non-comparable with the previous version (e.g. changing the GBM calibration window,
  redefining the APY-range anchors, adding a new asset class to the engine).
- **MINOR bump** (`v2.0 → v2.1`): an additive clarification, a new vault definition,
  or a parameter table update that does not break backward comparability.

A bump always requires a new ADR. Old versions remain on file for audit — LPs may
request the methodology version used to compute their entry NAV.

### 1.3 How to promote this draft

1. Product + Risk review and sign off on all sections.
2. Rename `v2.0-draft.md` → `v2.0.md` and update the Status line to `Active`.
3. Remove this ⚠️ DRAFT header.
4. Open an ADR (append to `docs/decisions/`) recording the promotion date and any
   changes made between this draft and the final text.
5. Update the `methodology_version` field in every future memo PDF to `v2.0`.
6. Existing memos generated before ratification retain `v1.0` as their reference —
   do not retroactively change them.

## 2. Monte Carlo Extension (V2 addition)

*Authorized by ADR-006. Rule-based engine from v1.0 remains the default.*

### 2.1 Purpose

The Monte Carlo (MC) mode provides a **probabilistic distribution** of projected APY
paths alongside the deterministic rule-based range from v1.0. It does not replace
rule-based output; both are shown side-by-side when MC is enabled. MC mode is an
explicit user opt-in.

### 2.2 Seed requirement (engine purity — non-negotiable #6)

- The PRNG seed is an **explicit engine input**. Same seed + same assumptions ⇒
  byte-identical output across runs.
- `Math.random()` ungoverned is **forbidden** inside `src/lib/engine/*`.
- `Date.now()` ungoverned is **forbidden** inside `src/lib/engine/*`.
- The seed, path count (N), and calibration window are recorded in every MC output
  object so any run is fully reproducible and snapshot-testable.

### 2.3 Default parameters

| Parameter | Default | Notes |
|---|---|---|
| Path count (N) | 1 000 | Configurable per run; minimum 500 for any published output |
| Calibration window | 36 months trailing `VaultSnapshot` + `MiningMetric` | Falls back to available history if < 36m |
| Horizon | 12 months | Same as rule-based default |
| Correlation matrix | BTC/difficulty estimated from historical log-returns | Recomputed on every calibration |

### 2.4 Stochastic model specification (GBM)

BTC price follows **Geometric Brownian Motion**:

```
dS = μ · S · dt + σ · S · dW
```

Where:
- `μ` (drift) = annualised mean of 36m daily log-returns from `VaultSnapshot.btcPrice`
- `σ` (volatility) = annualised standard deviation of the same log-returns
- `dW` = correlated Wiener increment, correlation ρ(BTC, difficulty) from historical data

Network difficulty uses a **mean-reverting drift with bounded step** calibrated on the
observed adjustment cadence from `MiningMetric`.

Hashprice is **never** an independent random variable. It is re-derived from each
sampled (BTC, difficulty) pair via the canonical `hashprice-formula.ts` function,
keeping MC consistent with live and backfill pipelines.

Energy cost is modelled as **contractual fixed + small normal noise** within the
partner contract band.

Stable/T-bill APY uses a **low-volatility normal** around the trailing 36m mean.

### 2.5 Calibration from VaultSnapshot historical data

```
Step 1 — Load trailing N-month rows from VaultSnapshot and MiningMetric
Step 2 — Compute daily log-returns for each stochastic variable
Step 3 — Estimate μ, σ, ρ (BTC/difficulty correlation matrix)
Step 4 — Store calibration metadata (window start/end, row count, any gaps)
Step 5 — Inject calibrated params into the engine as immutable config object
```

Gaps in the history (missing daily rows) are flagged in the confidence score
(see §2.7). If the effective calibration window drops below 6 months, MC output
is blocked and the engine falls back to rule-based only.

### 2.6 Reporting (p5/p50/p95)

For each of N paths:
1. Sample BTC, difficulty (correlated) over the horizon using the GBM/mean-reverting models.
2. Derive hashprice → mining yield → blended portfolio yield → `path_apy`.
3. Collect `{ path_apy }` into an empirical distribution.

Published output:

- **Headline APY range** (non-negotiable #1): `apy_low = p25`, `apy_high = p75`.
  **No single-point APY is ever published**, MC or not.
- **Distribution summary**: `p5`, `p50`, `p95` and `P(apy < floor)` for the vault's
  stated floor (e.g. `P(apy < 8%)`). These are clearly labelled as model output,
  never as a forecast.
- **Side-by-side display**: rule-based v1.0 range and MC range are shown together;
  they are never blended or averaged into a single figure.
- **Provenance badge**: `Estimated` (model), with seed + N + methodology link.

### 2.7 Confidence score (MC extension)

Inherits v1.0 confidence score unchanged, plus two additional degradation triggers:

| Trigger | Effect |
|---|---|
| Calibration history < 12 months | Degrades to Medium regardless of other factors |
| > 10% of paths outside 36m P5–P95 historical envelope | Degrades confidence by one level |

## 3. Multi-Vault Assumption Isolation

*Authorized by ADR-006. Vault id is a first-class key throughout the data model.*

### 3.1 Vault catalog (V2 scope)

| Vault id | Name | Status |
|---|---|---|
| `yield` | Hearst Yield Vault | Active (V1) |
| `defensive` | Hearst Defensive Vault | Buildable (V2) |
| `btc-plus` | Hearst BTC Plus Vault | Buildable (V2) |

### 3.2 Isolation rule

**Each vault carries its own methodology assumptions.** No vault may reuse another
vault's numbers silently. Concretely:

- Every parameter object passed to the engine (`EngineInput`) must include a `vaultId`
  field as a first-class key.
- Mining hashrate, revenue share, hedge ratio, energy cost, uptime SLA, and all other
  vault-specific parameters are stored and retrieved per `vaultId`.
- A `VaultSnapshot` row is scoped to exactly one vault; cross-vault aggregation is
  never performed unless explicitly labelled as a portfolio-level view.
- A projection for `btc-plus` must never use `yield`'s hashrate or assumptions — even
  if the values happen to be equal at a point in time.

### 3.3 Per-vault assumption tables

Each vault publishes its own assumption table in its spec file (`docs/spec/`).
The methodology v2.0 does not fix those values — it mandates their isolation.

A vault without a published assumption table cannot have projections generated.
The engine must throw if `vaultId` is unrecognised or assumptions are missing.

### 3.4 Provenance per vault

Every metric produced for a vault carries a provenance badge scoped to that vault's
data sources. A `Live` badge on `yield` does not imply `Live` on `defensive` — each
badge is independently earned from that vault's own data pipeline.

## 4. Share Classes

### 4.1 Overview

Each vault may offer up to two share classes: **Class A** (standard institutional)
and **Class B** (preferred / larger ticket). The classes differ on minimum ticket,
soft lock-up duration, and fee structure, which affects the **net APY range**
delivered to the LP.

### 4.2 Parameter comparison

| Parameter | Class A | Class B |
|---|---|---|
| Minimum ticket | $250,000 | $1,000,000 |
| Soft lock-up | 60 days | 30 days |
| Management fee (annualised) | 1.50% | 1.00% |
| Performance fee (above hurdle) | 15% above 8% hurdle | 10% above 8% hurdle |
| Redemption notice | 30 days | 15 days |

> Note: these parameters are subject to LP agreement. The methodology records the
> structure; actual terms are governed by the subscription documents.

### 4.3 Impact on net APY range

```
net_apy_range = gross_apy_range - management_fee - performance_fee_estimate
```

Where `performance_fee_estimate` is computed on the **midpoint of the gross APY
range** minus the hurdle rate, then scaled by the fee percentage. It is labelled
`Estimated` provenance and shown separately from the gross range.

- Class B LPs see a wider net range (lower fee drag) than Class A LPs.
- The methodology mandates that investor-facing outputs always show the **net APY
  range for the LP's specific share class**, never the gross range alone.
- Memos generated by the `investor-memo` agent must include the share class as a
  first-class field (`shareClass: "A" | "B"`) in their structured JSON output.

### 4.4 Share class + vault combination

The share class parameters above apply to all vaults in the catalog unless a vault's
spec file explicitly overrides them. Overrides are permitted at the vault level;
they must be documented in the vault's spec and referenced in the investor memo.

## 5. Provenance Ladder

*Unchanged from v1.0. Reproduced here for completeness and ease of reference.*

Every metric surfaced in the platform must carry one of the following badges, in
descending order of data quality:

| Badge | Meaning | Example |
|---|---|---|
| **Live** | Real-time or < 5 min old, direct from primary source | BTC price from Chainlink oracle |
| **Oracle** | From an on-chain or signed-data oracle, may be up to 1h old | Chainlink price feed, post 5-min SLA |
| **Attested** | Partner-signed attestation (monthly reporting cycle) | Mining uptime, energy cost |
| **Estimated** | Derived / modelled, not directly observed | MC distribution, projected APY |
| **Manual** | Human-entered value, no automated source | Override inputs, emergency fallback |
| **Stale** | Any source that has exceeded its freshness SLA | Any badge above after its TTL |

Rules:
- A metric inherits the **weakest** badge of all its inputs.
- When any input is `Stale`, the metric badge must be `Stale` regardless of other inputs.
- `Live > Oracle > Attested > Estimated > Manual > Stale` is the strict ordering.
- Downgrading a badge (e.g. `Live → Stale`) is automatic; **upgrading** requires a
  fresh data fetch — it is never done manually or by inference.

## 6. PTAI Format

*Unchanged from v1.0. Mandatory for all simulations and rebalancing actions.*

Every rebalancing recommendation, scenario output, and simulation result must follow
the **PTAI** structure:

```
Projection  — What the model expects to happen (range, not point estimate)
Trigger     — The condition that would warrant action
Action      — The specific rebalancing or operational step recommended
Impact      — Quantified effect on APY range, risk profile, and provenance
```

Rules:
- PTAI is the only permitted output format for the `scenario-narrative` and
  `risk-explanation` agents.
- Auto-execution of rebalances remains **forbidden** (ADR-006). PTAI outputs are
  always human-reviewed before any action is taken.
- The "Action" field must never contain a commitment to future returns. Use ranges
  and conditional language ("would improve APY range by approximately X–Y bp").
- The "Impact" section must include an updated provenance badge reflecting the
  projected state after the action.

## 7. Forbidden Vocabulary

The following words and phrases are **forbidden in all agent outputs**, memos,
projection narratives, and investor-facing text. Code enforcement is via the
`forbidden-words-linter.ts` module, which runs on every agent output before
persistence.

### 7.1 Forbidden list

| Term | Forbidden form(s) |
|---|---|
| Guarantee | `guarantee`, `guaranteed`, `guarantees` |
| Promise | `promise`, `promises`, `promised` |
| Certain | `certain`, `certainly`, `certainty` |
| Will deliver | `will deliver`, `will generate`, `will produce` (in forward-looking context) |
| Risk-free | `risk-free`, `riskless`, `zero risk`, `no risk` |

### 7.2 Permitted exception

The phrase **"not guaranteed"** is explicitly **permitted and encouraged** in
disclaimers, confidence score narratives, and assumption blocks. It reinforces
regulatory compliance and must appear in every LP-facing projection output.

Example of correct usage:
> "Projected APY range of 9.4–12.8%. Past performance does not guarantee future
> results. Returns are not guaranteed and depend on stated assumptions."

### 7.3 Enforcement

- The linter runs synchronously on the raw string output of every agent before the
  result is written to the database.
- A match causes the agent call to return an error with `code: "FORBIDDEN_WORD"` and
  the offending term. The caller must not surface partial output to the UI.
- CI runs the linter against all fixture outputs in `src/lib/agents/__tests__/`.

## 8. Disclaimers

### 8.1 Standard projection disclaimer (all modes)

> Projections are conditional on stated assumptions. Past performance does not
> guarantee future results. Hearst vaults are offered exclusively to professional /
> qualified investors via a Cayman Exempted Limited Partnership. Subject to minimum
> subscription, soft lock-up, and jurisdictional restrictions. Not an offer or
> solicitation where prohibited.

### 8.2 Monte Carlo addendum (MC mode only)

> Monte Carlo results are model output conditional on stated volatility and
> correlation assumptions calibrated on historical data. They are not a forecast or
> a commitment of future returns. Percentiles describe the model's distribution, not
> the probability of actual outcomes. Returns are not guaranteed.

### 8.3 Multi-vault addendum

> Each vault operates under its own assumption set and share class structure.
> Performance of one vault is not indicative of the performance of any other vault.
